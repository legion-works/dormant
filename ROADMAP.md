# Roadmap

Direction for `dormant` ŌĆö the OLED-preserving presence daemon. Grouped by state, not by date, and within each group ordered by priority (highest first). Items move down as they ship; nothing here is a dated promise. Small concrete fixes and polish live in the [issue tracker](https://github.com/legion-works/dormant/issues), not here.

**v0.1.0 shipped 2026-07-09** ŌĆö [release + installers](https://github.com/legion-works/dormant/releases/tag/v0.1.0). The core daemon, all three control surfaces, and the blanking paths run on the maintainer's AOC AGON AG326UZD and Samsung S90D. `master` holds the last released state; `dev` is the integration branch.

## Shipped

- **Daemon core** ŌĆö config schema + strict validation (unknown-key rejection, cross-reference checks), zone fusion engine (`any`/`all`/`quorum`/`weighted`), rules engine, per-display state machine, hot reload with phase carry-over, single-instance `flock` guard. Fail-safe presence throughout: data loss makes a sensor `unavailable`, never `absent` ŌĆö a room you can't see is never blanked blind.
- **Sensors** ŌĆö MQTT (with native per-broker auth), Home Assistant WebSocket, USB-serial LD2410 mmWave. One module per backend; ESPHome sensors drop in over the existing MQTT path with no new code.
- **Retained values and availability** ŌĆö MQTT retained values flow through on subscribe and reconnect. `sensors.<id>.availability_topic`, `availability_payload_online`, and `availability_payload_offline` support non-Zigbee2MQTT LWT conventions. The `reported` diagnostic distinguishes "no data since start" from a sensor that later went offline. A startup/reload warning flags MQTT zones using `zones.<id>.unavailable_policy = "absent"`.
- **Display controllers** ŌĆö `ddcci` (VCP `0xD6` power-off and brightness-zero), `samsung-tizen` (`KEY_PICTURE_OFF`, power-key, Wake-on-LAN, network pairing, plus an IP-Control G2 backlight-dim path on port 1516), `kwin-dpms`, `ha-passthrough`, and a generic `command` escape hatch. Per-display fallback chain with bounded wake-retry. The Samsung control connection tracks the TV's own heartbeat and reconnects before a send rather than firing keys into a silently-dropped idle socket.
- **Audio-preserving blank modes** ŌĆö DDC/CI `0xD6` blanks capable monitors without tearing down the OS output. Samsung Tizen picture-off leaves TV audio running but pauses the HDMI source; port-1516 `brightness_zero` keeps the source and its audio running while dimming the panel near-black. DPMS remains the fallback where output audio does not matter.
- **Render ladder** ŌĆö a local Wayland layer-shell overlay: black surface as the final blank fallback, and a libmpv-driven screensaver (folder/URL playlists, `fill`/`fit`/`stretch`/`center` scaling, crossfade transitions, muted by default). Escalation ladders blank ŌåÆ dwell ŌåÆ power off on a configurable schedule.
- **Pixel shift** ŌĆö the screensaver surface shifts 2 px every 2 minutes by default; `displays.<id>.screensaver.shift_px = 0` disables it. The uniform black overlay never shifts.
- **Panel-wear tracking** ŌĆö brightness-weighted on-hours, VCP `0xC0` seeding where available, local JSON ledgers, `GET /api/wear`, the web panel-exposure card, `wear.*` settings, and `displays.<id>.panel_type`. v1 attribution is panel-wide and advisory; it does not enforce rest windows or track regions.
- **Control surfaces** ŌĆö `dormantctl` (status, pause/resume, blank/wake, reload, validate, watch, doctor, pair, emergency-wake), a loopback-only web dashboard (live state over WebSocket, doctor view, two-tab config editor with an atomic validated apply pipeline), and a KDE `StatusNotifierItem` tray with per-display submenus.
- **Emergency wake** ŌĆö `dormantctl emergency-wake` force-wakes every display and pauses automation. It tries daemon IPC first; when IPC fails, it builds the controllers directly and drives every wake in parallel.
- **Control-path verification** ŌĆö `dormantctl doctor exercise <display>` blanks and wakes the panel, then reads state back where the controller supports it. The exercise runs through the daemon, pauses the display's rule for the window, and always restores wake state.
- **Manual-only displays** ŌĆö a display in `[displays]` referenced by no rule is hand-controllable and never auto-blanked. This is how a TV joins dormant without a keep-awake dummy zone.
- **Doctor** ŌĆö hardware/connectivity probes for config, MQTT, HA, USB, DDC/CI, and Samsung (reachability, power state, token), plus the exercise mode above.
- **Flat footprint** ŌĆö the Tokio worker pool is capped to two threads, `malloc_trim` runs after every screensaver teardown, and the systemd unit sets `MALLOC_ARENA_MAX=2`. Together these drop the post-screensaver RSS floor from ~265 MB to ~90 MB; idle RSS stays flat across blank/wake/screensaver/reload cycles.
- **Reload-window integrity** ŌĆö control messages and presence events forwarded during a config-reload generation swap are retried across the swap instead of being dropped into the torn-down generation. Closes the narrow lost-command window on both the control and presence paths ([#9](https://github.com/legion-works/dormant/issues/9)).
- **Failure notifications** ŌĆö repeated wake failures and exhausted blank controller chains appear as critical desktop notifications, a tray `Failure` state, and a web-dashboard banner. `notifications.*` controls the desktop notices; tray and web failure state remain active when they are disabled.
- **Delivery** ŌĆö CI matrix covering fmt, clippy pedantic, tests, MSRV, Linux/macOS/Windows portability, license/advisory audits, gitleaks, TOML/spell/docs gates, and mdBook; local `lefthook` hooks mirroring the same gates; cargo-dist release pipeline producing tarballs + shell installers for x86_64/aarch64 Linux and, since the macOS M1 milestone, arm64/x86_64 macOS ŌĆö with a release-artifact smoke test on three of those targets (x86_64 Linux plus both macOS architectures, including a `plutil -lint` check on the packaged `LaunchAgent` plist; aarch64 Linux is built but has no smoke leg ŌĆö no arm64 Linux runner is wired up) gating publish.
- **Watchdog + last-known-good rollback** ŌĆö health-gated LKG snapshots, boot rollback for invalid or crash-looping configs, and a `Type=notify` systemd unit with `WatchdogSec=150`. Recovery after a rollback boot currently requires `systemctl --user restart dormant` after fixing the config ([#53](https://github.com/legion-works/dormant/issues/53)).
- **Web entity CRUD and Samsung pairing** ŌĆö the config editor creates and deletes sensors, zones, displays, and rules when `daemon.entity_crud_enabled = true` (default). The web pairing wizard and `dormantctl pair samsung <host>` store Samsung tokens in `credentials.toml`; `daemon.pairing_enabled` gates the web route.
- **Audio- and call-aware blanking** ŌĆö a `pw-dump`-polling PipeWire inhibitor makes `"audio-playback"` and `"call"` rule literals functional: a running output stream or an active call holds a display awake, independently of and combinable with the user-activity inhibitor. Global `[audio]` settings (`poll_interval`, `min_active`, `call_roles`, `playback_roles`, `capture_is_call`, `pw_dump_command`) tune classification; `capture_is_call` defaults to `false` to avoid false positives from idling mic-capable apps. Fails toward blanking on any probe error.
- **macOS (M1)** ŌĆö native macOS support, arm64 and x86_64: DDC/CI display control (shared `ddcci` controller, vendored `ddc-macos` fork), the `macos-gamma-black` audio-safe Quartz gamma-table blank controller with a daemon-independent emergency-restore breadcrumb, the `macos-display-sleep` whole-machine `pmset` fallback, a CoreGraphics idle source, read-only `dormantctl doctor macos-idle` / `macos-display-sleep` / `macos-power` diagnostics, and a `launchd` `LaunchAgent` (`dormantctl launchd install`/`uninstall`) with cargo-dist release artifacts for both macOS targets. `dormant-tray` is packaged for macOS but **not functional there** ŌĆö a KDE `StatusNotifierItem` applet has no macOS equivalent yet, and there is no macOS watchdog-parity (wedged-daemon) detection the way systemd's `WatchdogSec` provides on Linux.

## Near-term

- **Spatial wear attribution + wear-evening screensaver** ŌĆö extend the shipped panel-wide ledger into per-region attribution and panel-type-specific weighting, then use content placement to avoid repeatedly lighting the same regions. Pixel shift is already shipped; enforced rest windows are not. See the [prior-art survey](docs/research/2026-07-09-oled-health-prior-art.md) and [design + probe findings](docs/research/2026-07-09-oled-health-design.md).
- **Public-repo polish** ŌĆö dashboard/tray screenshots in the README, a release badge, a web UI version label sourced from the crate version, and a CI smoke job that installs the built release artifact and runs `dormantd --validate-only`.

## Planned

- **Scenario record / replay + hardware-simulation harness** ŌĆö capture real sessions (sensor events ŌåÆ decisions) as replayable fixtures for the offline rules engine, and swap sensor/display backends for deterministic simulators in CI. Lets a contributor validate a config or a change against real and simulated scenarios without risking physical displays ŌĆö directly attacks the single-maintainer verification bottleneck. `dormantctl scenario record|replay`.
- **BLE phone-as-key presence** ŌĆö passive Bluetooth-LE scanning on the host for an allow-listed phone, watch, or earbud, with per-room RSSI thresholds feeding the zone engine. Uses the host radio; no broker or purchased sensor required.
- **Webcam vision fallback** ŌĆö an opt-in, locally-processed motion/presence detector using the host webcam (OpenCV-class, no network, no storage, no frames ever leave the machine) as a fallback presence source when the broker is down. A camera is a sensitive sensor even when local, so this ships off by default behind an unmissable opt-in with plain documentation of exactly what it sees and where the data goes (nowhere) ŌĆö consistent with the no-telemetry non-goal.
- **mDNS discovery and pairing** ŌĆö dormantd instances advertise themselves on the LAN (zeroconf/`_dormant._tcp`) and discover each other; `dormantctl pair <host>` and the web wizard get a discovered-instances list instead of hand-typed addresses. One trusted-instance handshake (token exchange, stored in `credentials.toml`) establishes the peer relationship the KVM-switch and multi-instance features below build on. Loopback-first ethos holds: discovery is LAN-scoped, opt-in, and never leaves the local network.
- **KVM-switch mode** ŌĆö two dormantd instances on separate machines driving the same multi-input display act as a software KVM switch: a hotkey (or a rule on user input ŌĆö the machine whose keyboard/mouse wakes claims the panel) flips the display's input source via DDC/CI VCP `0x60` input-select, with the yielding instance blanking/releasing and the claiming instance waking ŌĆö the `OwnershipGate` seam arbitrating who owns the panel. Switch events can run configured hook commands (`on_acquire`/`on_release`) so peripherals follow the display: trigger a USB switch, a SwitchBot, or any custom script. Builds on mDNS pairing above for peer discovery/coordination; the maintainer's AOC + Linux-desktop + Mac setup is the reference hardware.
- **macOS status bar item** ŌĆö the tray's macOS counterpart: a native menu-bar extra (icon + menu + widget) with the same surface as the KDE tray ŌĆö per-display state, pause/resume, blank/wake, emergency wake, web UI link. Closes the "dormant-tray is packaged but nonfunctional on macOS" gap called out in the M1 shipped notes; the macOS M2 milestone's tray task is this item.
- **Global hotkeys** ŌĆö bind a key to blank, wake, pause, or emergency-wake (the shipped panic command is the priority binding) as a first-class feature: the daemon registers shortcuts through the compositor's path (the XDG `GlobalShortcuts` portal or KDE's KGlobalAccel ŌĆö Wayland has no raw global grab), with a config block to declare them. Users can already bind their own shortcuts to `dormantctl` commands today; this makes it declarative.
- **Doctor-assisted issue drafting** ŌĆö `dormantctl doctor` already gathers the exact hardware, environment, and probe output the issue templates require. Let it write a ready-to-file bug report or feature request to disk, pre-filled with display model + connection, OS/compositor/session, controller, and probe results, captured at the moment something fails so it can be filed later without hand-reconstructing the context. `doctor --report-issue` / `--draft-feature`.
- **More display controllers** ŌĆö HDMI-CEC (via `libcec` + a CEC adapter ŌĆö unlocks most TVs regardless of brand, and can report the active source for input-aware control), LG webOS (network TVs), Gnome DPMS (audio-safe where the output has no sound).
- **Schedule and conditional rules** ŌĆö time-of-day and quiet-hours conditions on rules: "never blank between 7pm and 11pm", "only auto-blank after midnight". Presence stays the trigger; time becomes a gate.
- **Packaging** ŌĆö an AUR package first (the maintainer's own distro ŌĆö dogfoods the release artifacts), then `.deb` / `.rpm`, a Flatpak for the web UI + tray, and systemd units shipped inside the release tarballs so an install isn't complete only after hand-copying unit files.
- **Local observability** ŌĆö an opt-in, loopback-only metrics/history surface: a blank/wake/screensaver event timeline and per-display uptime, viewable in the web UI (and optionally a Prometheus endpoint for the homelab crowd). Strictly local, opt-in, never phoned home ŌĆö see Non-goals.

## Exploratory ŌĆö not committed

- **Energy-savings ledger** ŌĆö estimate power, cost, and COŌéé saved from blank events (blanked duration ├Ś nominal display wattage) and surface it in the dashboard. Parked until the event-history surface can supply reliable duration data.
- **Presence as a local API** ŌĆö expose fused zone presence over loopback MQTT/REST and accept commands over MQTT so other local services can consume room state or drive dormant headlessly. The blanking daemon remains the primary scope.
- **Input-aware display control** ŌĆö use the active input source to pick a local controller (DDC/render) when the PC owns the panel and a remote controller otherwise. Parked: the maintainer's S90D exposes no local input signal, so it needs a multi-input DDC monitor, an LG webOS TV, or an HDMI-CEC adapter to be worth building ŌĆö the CEC controller above may unblock it. The `OwnershipGate` seam is already in place for it.
- **Multi-instance coordination** ŌĆö several dormant instances arbitrating one shared display over MQTT, so a laptop and a desktop don't fight the same TV. Rides the same `OwnershipGate` seam.
- **macOS tray parity and Windows** ŌĆö macOS itself moved out of this section (native display control shipped in the M1 milestone ŌĆö see Shipped above). What remains exploratory: a `dormant-tray`-equivalent status/control surface for macOS (menu bar item; no committed design or timeline), and Windows, where the codebase cross-compiles today (portability CI is green) but native display control is unbuilt.

## Non-goals

- **No telemetry, no phone-home, ever.** The daemon talks only to the sensors and displays you configure. Local-only features (webcam vision, observability) stay local by construction and opt-in.
- **No cloud dependency.** Everything runs locally; network controllers reach your own devices on your own LAN.
- **No weakening of fail-safe presence or the wake path.** A screen that won't wake is the worst failure mode; correctness there is never traded for a feature.
